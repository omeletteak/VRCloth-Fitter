# エコシステム構想: フィット情報の標準化

このドキュメントは**構想資料**です。プリフライト診断([DESIGN.md](DESIGN.md) §9)を個人用ツールからエコシステムの互換性標準へ育てる道筋を記録します。実装を拘束するのは DESIGN.md と ROADMAP のみで、本書の内容は採用時にそちらへ昇格させます。情報設計の原則は [INFORMATION_ARCHITECTURE.md](INFORMATION_ARCHITECTURE.md) を参照。最終更新: 2026-06-13。

## 0. 一行で

**採寸表が予測し、診断が保証し、修正が残差を吸収する。** 形状データはどこにも動かない。

## 1. 背景: 「対応アバター」モデルの構造的なコスト

現在の衣装市場は、ショップがアバターごとに手作業でフィッティングした「対応版」を作る方式が主流。このため:

- 衣装1着の市場規模は対応リストの長さで頭打ちになる
- 対応外アバターの所有者は購入の判断材料がない(目視の試着頼み)
- ショップは人気アバター全てに対応し続ける工数を負う

現実のアパレル産業は同じ問題を「サイズ表 + S/M/L + お直し」で解いてきた。本構想はその情報構造の輸入であり、**貫通修正が「お直し」の役**を担う。

## 2. 三層構造

| 層 | 役割 | 動く情報 | 実装 |
|---|---|---|---|
| **予測** | 購入前に「どのサイズが合うか」を判断 | 採寸表(周径スカラー十数個) | 採寸機能(§5) |
| **保証** | 着せた状態で「枠内か」を測定・判定 | Fit Report(統計値+判定) | プリフライト診断(実装済み) |
| **吸収** | 残差(cm 級・半径方向・局所)を自動修正 | なし(ローカル処理・No Cache) | 貫通修正パイプライン(実装済み) |

三層が成立するのは [DESIGN.md](DESIGN.md) §9 のサポート範囲内に限る: 残差が浅いからこそ予測が周径スカラーで済み、修正が局所変位で済む(この同一性の議論は [INFORMATION_ARCHITECTURE.md](INFORMATION_ARCHITECTURE.md) §4)。

## 3. 買い手側: 購入前にフィットを知る3チャネル

1. **ショップ公開の計測マトリクス** — 対応一覧の拡張として「公式対応外だが計測済み」の判定と数値を載せる。ショップは対応制作のため主要アバターを所持しており、計測の増分コストはほぼゼロ
2. **コミュニティ共有** — Fit Report は復元不能な集計スカラーなので、アバターと衣装を両方所持する誰もが権利不安なく数値を公開できる。ロングテールはここが埋める
3. **試着版 + ローカル診断**(本命)— BOOTH の試着版配布の慣行に乗る。試着版を自分のアバターに着せてプリフライトを回せば、**誰も何も公開しなくても任意のアバターで購入前に実測できる**。既存の「試着して目視」が「試着して計測」に格上げされる

### 片側性の補完: クリアランス統計

貫通修正は「体が服より大きい」方向しか扱わないため、現状の診断に**ブカブカは映らない**(検出ゼロの完全な緑 ≠ 似合う)。サイズ選択の指標として完結させるには、逆方向の統計 — 体表面から布までの隙間(クリアランス)の percentile — を Fit Report に加え、判定を両側化する:

- **TIGHT**(赤: 枠外の食い込み)/ **FIT**(緑〜黄: 修正で吸収可能)/ **LOOSE**(干渉ゼロだが隙間過大)

実装は検出と同じスキャンの符号反対側の集計であり軽量。正しいサイズの読み方は「最も緑」ではなく「**TIGHT でない最小サイズ**」になる。

## 4. ショップ側: サイズタイリングと QA

### サイズタイリング

修正可能包絡が深さ ~1cm 級(§9)なので、**主要周径で ~2cm 刻みのサイズ間隔**にすれば、どのアバターも必ずどれかのサイズから 1cm 以内に落ちる。体型ファミリー内なら S/M/L の3サイズで、現行の十数アバター個別対応ぶんをカバーする計算になる。

制約:
- **比率ファミリーは跨げない** — ちび系は標準頭身の「小さいサイズ」ではなく別の比率クラス(§9 の接線限界)。タイリングは「ファミリー数 × サイズ数」
- **ポーズ依存・審美は未計測** — 静的ポーズ × カプセルの診断は、動かした時のフィットや「裾がどこに落ちるか」という手作業対応の品質を代替しない(フェーズ3の代表ポーズ・半径推定で信頼度が上がる)

### 出荷 QA としてのヌルテスト

公式対応版のリリース前に対応アバターでヌルテスト(E2E_TEST_GUIDE.md §2)を回し「緑」を確認する — 今日の機能のままショップが**単独で**得をする使い方。ここでツールがショップのワークフローに入ることが、後段(計測結果の公開)への自然な入口になる。

### 経済的な動機づけ

ショップへの提案は「対応工数の削減」(守り)より「**市場拡大**」(攻め): 公開した黄〜緑の一行ごとに、その衣装の購入可能者が未対応アバターの所有者へ広がる。SKU を増やさず市場が広がる。バッジ(例: 「VRCloth-Fitter Ready」)はその可視化装置。赤判定時に体型変換ツールへ誘導すれば、リターゲティング製品とは競合でなく補完という位置取りを製品の挙動として示せる。

## 5. 採寸表: 予測層の仕組み

- **標準計測点**: Humanoid ボーンを解剖学的ランドマークとして 10〜15 点を定義(例: バスト = Spine〜UpperChest 帯の最大周径、ウエスト = 同帯の最小周径、太もも = UpperLeg 25% 地点の周径、肩幅、首回り…)。素体メッシュをボーン軸に垂直にスライスし、断面の凸包周長を「メジャーを回した値」として測る
- **権利安全性**: 周径スカラー十数個から彫形は復元できない。現実のアバター商品ページが身長を載せているのと同列の情報粒度(詳細は INFORMATION_ARCHITECTURE.md)
- **衣装側**: ショップが同じ計測点の「仕上がり寸法」(衣装内周)を公開する。現実のアパレル EC の着丈・身幅表記と同じ慣行
- **診断との定量リンク**: 周径の不足 Δc は半径の不足 Δc/2π に対応する。「バスト周りで 6cm 足りない」≒「深さ約 1cm の貫通」— サイズ表の数字と §9 のしきい値が同じ物差しに乗り、採寸表から診断結果(最大深度)を予測できる
- **実装の合流**: 周径計測はフェーズ3「カプセル半径自動推定」と同じ機構(肢のスライス半径 ≈ 周径/2π)。一つの実装が採寸と半径推定の両方を担う

## 6. Fit Report: 保証層の流通形式

- **機械可読**: JSON 一枚 — ツールバージョン、しきい値バージョン、計測条件、アバター/衣装の識別(名前+**メッシュハッシュ**: どのバージョンの資産に対する計測かを固定する。ハッシュは形状を漏らさない)、renderer ごとの統計、判定
- **計測条件の固定**: margin 5mm・デフォルトポーズ・シェイプキー 0 など、ISO の測定条件のように規格化して報告書に刻む
- **人間可読**: 緑/黄/赤バッジ画像へのレンダリング(BOOTH ページ用)
- **信頼は再現性で**: 計測は決定的で安価なので、両方の資産を持つ者は誰でも再計測して検証できる。偽装は自滅的。加えて**再配布自由な無料アバター数体を「基準マネキン」**に指定すれば、有料資産なしで誰でも検証できる共通の物差しになる

## 7. NDMF パス化: 吸収層の製品形態

ショップが「3サイズ + VRCloth-Fitter 前提」と言えるのは買い手の手間がゼロのときだけ。NDMF のビルドパスにすれば、着せてアップロードするだけで修正が走り、摩擦が消える。

技術的にも本質的: NDMF パスは **Merge Armature の後**に実行されるため、エディタ上の「素体と衣装のアーマチュアが別物」問題(E2E_TEST_GUIDE.md §5 の偽貫通)が構造的に消え、「Merge Armature 後の貫通修正」というプロジェクトの定義に正確に一致する実行タイミングになる。エディタのボタンは試着・プレビュー用、NDMF パスが製品形態。

## 8. 普及の階段

標準は「みんなが使うと便利」では広まらない。「一人で使っても便利」な段階を積む:

1. **個人の修正ツール**(フェーズ1 = 現在)— 単独で完結する価値
2. **ショップの出荷 QA**(機能は今日のまま)— ショップが単独で得をする
3. **計測結果の公開**(Fit Report + クリアランス)— QA の数字を載せるだけ。増分コストほぼゼロ
4. **採寸表 + サイズ設計**(タイリング)— エコシステムとしての完成形

各段は前段の自然な延長で、どの段で止まってもそこまでの価値が残る。

## 9. 未決事項

- 基準マネキンの選定(再配布自由・体型空間を張る数体)
- しきい値・判定のバージョニング運用(較正で変わる判定の互換性管理)
- 標準計測点の正式定義(→ MEASUREMENT_SPEC として分離予定)
- Fit Report の JSON スキーマ(→ FIT_REPORT_FORMAT として分離予定)
- クリアランス(LOOSE 側)のしきい値

## 10. ガードレール

本構想はフェーズ1を変更しない。実機 E2E でのしきい値較正は、この構想の文脈では「v1 の物差しの校正」という意味を持つ — 公開に耐える数値はそこから始まる。実装順: E2E 較正 → クリアランス統計 → 採寸(=半径推定) → Fit Report フォーマット → NDMF パス。

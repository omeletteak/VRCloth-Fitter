# 情報アーキテクチャ: 何を運び、何を運ばないか

このドキュメントは**構想資料**です。[DESIGN.md](DESIGN.md) §4(変換プロファイルの所見)と §5(No Cache)で確立した原則を、エコシステム構想([ECOSYSTEM_VISION.md](ECOSYSTEM_VISION.md))まで拡張したときの情報設計を記録します。最終更新: 2026-06-13。

## 0. 一行で

**形状は動かさない。動かすのは、形状に戻せない判断材料だけ。**

## 1. 変換と判定では、要求される情報量の格が違う

衣装フィッティングに関わる処理は、必要とする情報の量で2つに分かれる。

- **変換**(リターゲティング)— 衣装を体型 A から体型 B へ変形するには、A と B の形状(または差分場)が全情報として必要になる。変換は可逆性を要求するため、扱うデータは原理的に**復元可能**(reconstructive)になる。DESIGN.md §4 で確認した「変換プロファイルが素体点群を実質的に丸ごと含む」現象は、実装の不注意ではなく、変換という仕事の情報要求量に由来する構造的なもの
- **判定**(フィットの予測・検証)— 「このサイズで合うか」の答えは実質数ビットであり、入力も集計値で足りる。判定に使うデータは**非可逆な要約**(decision-grade)にできる

本プロジェクトは問題を「予測(判定)」と「修正(ローカル処理)」に分解した。修正には現物が必要だが、現物=アバターと衣装は着せ替えの時点で既にユーザーの手元に揃っている。したがって**機械や当事者の境界を越えて動く必要があるのは判定用の情報だけ**であり、それは復元不能な要約で済む。

## 2. 品目表: 動くもの・動かないもの

| 動くもの | 誰→誰 | 中身 | 復元可能性 |
|---|---|---|---|
| アバター採寸表 | アバター作者または所有者 → 公開 | 周径など十数個のスカラー | 彫形は復元不能 |
| 衣装の仕上がり寸法 | ショップ → 公開 | 同じ計測点の十数個のスカラー | 衣装メッシュは復元不能 |
| Fit Report | 両方の所有者 → 公開 | 統計値十個程度+判定+バージョン刻印 | 復元不能 |
| 判定(TIGHT/FIT/LOOSE) | ツール → ユーザー | 実質数ビット | — |
| 衣装本体 | ショップ → 購入者 | 通常の販売 | (取引そのもの) |

| 動かないもの | 理由 |
|---|---|
| 素体メッシュ・点群 | No Cache(DESIGN.md §5) |
| 変位場・差分キャッシュ | セッション内で生成し破棄。保存も出力もしない |
| 衣装の設計情報(型紙相当) | ショップの IP。本構想はその開示を一切要求しない |

形状級のデータで境界を越えるのは販売される衣装そのものだけで、それは取引の本体である。

## 3. 先例: 現実のアパレルは100年前からこの形

この情報設計は新発明ではなく、現実の服飾産業が物理的制約(身体は複製して送れない)の下で発達させた流通形式の輸入である。

- ブランドは**型紙を渡さない。サイズ表を渡す**
- 客は**ボディスキャンを渡さない。採寸値と S/M/L の選択を渡す**
- 試着室から持ち出されるのは「合った/合わなかった」という判定だけ

ヌード寸法(身体)と仕上がり寸法(衣服)の対で互換性を判断し、残差はお直しが吸収する — ECOSYSTEM_VISION.md の三層はこの構造の写像であり、貫通修正は「お直し」に対応する。

## 4. サポート範囲と情報境界は同じ線

[DESIGN.md](DESIGN.md) §9 のサポート範囲(深さ cm 級・半径方向・局所)と、「統計で予測できる範囲」は同じ境界線である。

- 残差が浅いからこそ、(a) 修正が局所変位で済み、(b) 予測が周径スカラーで済む(周径差 Δc ≈ 半径差 2πΔr の線形対応が成り立つ)
- 体型差が深くなると、(a) 再ドレープ=形状処理が必要になり、(b) 予測にも形状情報が必要になる

**情報要求量と変形の難しさは同じパラメータで一緒に増える。**枠外(リターゲティング)を担う処理が形状データを必要とするのはこのためで、枠内に留まる本プロジェクトが集計値で済むのもこのためである。No Cache は「制約をうまく回避した」結果ではなく、浅い残差の問題を選んだ時点で自動的に手に入る性質だった。

## 5. スケール則: 集めるほど安全になる側を選ぶ

二つのアーキテクチャは、データ集積の規模に対して逆向きの性質を持つ。

- **形状データの集積**は、規模に比例して権利上の慎重さを要求する(含まれる第三者の形状が増えるため)。運用(権利者自身による配布など)での緩和が前提になる
- **判断材料の集積**は、規模に比例して較正精度と検証可能性が上がる(レポートが増えるほどしきい値の物差しが正確になり、再計測できる検証者も増える)

本プロジェクトが蓄積すべき資産は、データの量ではなく**物差しそのもの**である:

1. **較正データ** — Fit Report と「実際に綺麗に直った/ダメだった」のラベル。集計値なので公開プールにできる
2. **計測定義** — 標準計測点の置き方。現実の採寸の流儀(JIS 等)から借りられる
3. **信頼** — 決定的な計算、バージョン刻印、再現による検証可能性、監査可能なオープン実装(MIT)

標準は何も所有しないからこそ、誰のものでも測れる。

## 6. 公開運用上の規律

この構想自体を公開リポジトリで育てるにあたっての規律:

- 競合製品への言及は DESIGN.md §3〜§4 と同じ調子を保つ — 公開情報に基づく事実と、設計上の教訓のみ。アーキテクチャの違いは構造の問題として書き、製品や作者の評価として書かない
- 第三者(ショップ・アバター)の名前は事実関係(対応関係・テスト対象)の記述に限る
- リポジトリ public 化の前に、docs 全体と `.beads/issues.jsonl` のノート内容を上記の基準で一度レビューする(ROADMAP フェーズ2)

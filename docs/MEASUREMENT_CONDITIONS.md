# 計測条件と採寸の誤差モデル (MEASUREMENT_CONDITIONS)

このドキュメントは**構想資料**です。[MEASUREMENT_SPEC.md](MEASUREMENT_SPEC.md) §6 で「保留」とした*条件の完全化*を概念として昇格させ、採寸の「誤差」とは何か、そして素体が連続変形できるがゆえの組み合わせ空間をどう畳むかを主題化します。実装を拘束するのは [DESIGN.md](DESIGN.md) と [ROADMAP.md](../ROADMAP.md) のみ。関連: [MEASUREMENT_SPEC.md](MEASUREMENT_SPEC.md)(何を測るか)、[DIAGNOSTIC_HONESTY.md](DIAGNOSTIC_HONESTY.md)(計器誤差と信頼区間)、[CORRECTABILITY_FIELD.md](CORRECTABILITY_FIELD.md)(f を場として持つ)、[FAMILY_MODEL.md](FAMILY_MODEL.md)(採寸空間・予測と門番)。最終更新: 2026-06-27。

## 0. 一行で

**採寸値 = f(メッシュ, 計測条件)。機械の座標計算は決定論的＝計算誤差はゼロ。「誤差」と呼べる唯一のものは、記録した計測条件と実運用条件の不一致(condition mismatch)であり、その源は人間の条件設定(シェイプキー・active 構成)である。** 素体は連続変形できるので計測条件は空間をなす ── だが連続だからこそ、組み合わせ爆発は「関数化＋1点評価＋予測/門番の二層」で畳める。

## 1. 誤差の正体は condition mismatch

人間の身体計測と違い、3D 素体の採寸はセンサーを通さない。同じメッシュと同じ条件を入れれば、機械は必ず同じスカラーを返す ── **確率的な計測誤差は存在しない**。したがって採寸に「誤差」があるとすれば、それは値のばらつきではなく、**どの状態を測ったかの取り違え**である。

> 採寸値 = f(メッシュ, 計測条件)、計測条件 = { シェイプキー状態, ポーズ, スケール正規化, どのレンダラーが active か, … }
>
> **誤差 ≡ 記録された計測条件 ≠ 実運用条件(condition mismatch)。**

この誤差モデルは [DIAGNOSTIC_HONESTY.md](DIAGNOSTIC_HONESTY.md) の「計器誤差→信頼区間」とは別の層にある。あちらは*半径=円断面近似*のような**推定の不確かさ**(同じ条件でも近似ゆえに真の周径とずれる)。本書は**条件の取り違え**(近似が完璧でも、違う状態を測れば違う値が出る)。両方が独立に効くので、採寸の正直さは「信頼区間(推定の幅)＋条件スナップショット(何を測ったか)」の二点セットで担保される。

## 2. 二つの採寸 ── 実効採寸と共有採寸

「採寸」と一語で呼ぶが、用途で二つに分かれ、condition mismatch の効き方が正反対になる。

### 2.1 実効採寸(self・衝突判定用) ── 既に mismatch ゼロ

declipper が自分の貫通修正に使う採寸。**実運用条件そのものを見るので、構造的に mismatch が起きない**。コードで裏が取れている:

- `VRClothMeshCapture.cs` の `BakeMesh(baked, false)` は**シェイプキー・ポーズ適用後**の頂点を焼く。バストをシェイプキーで縮めてから着せれば、縮めた素体がそのまま衝突体になる。
- 同 `Capture` は `!activeInHierarchy || !renderer.enabled` を**スキップ**する。胸メッシュを inactive にして小さい方だけ active にすれば、active な方だけが採寸・衝突対象になる。EditorOnly はビルド時にストリップされ、パスが走る前に消えている。
- declipper は NDMF パスで Merge Armature の**後**に走る([NDMF_INTEGRATION_SPIKE.md](NDMF_INTEGRATION_SPIKE.md))。見ているのは「アップロード直前の実状態」。

**「ビルド時の実状態を見る」という設計それ自体が condition mismatch を不能にしている。** 実効採寸に誤差が入る隙は原理的にない。

### 2.2 共有採寸(ecosystem・マッチング用) ── 条件刻印が必須

他者と持ち寄って比較する採寸([MEASUREMENT_SPEC.md](MEASUREMENT_SPEC.md) §5 コーパス・§7 マッチング行列)。ここで mismatch が牙を剝く。現状は**基準が二つ混在**している:

| 出力 | 取得条件 | 性質 |
|---|---|---|
| `meshHash`(`MeshFingerprint`) | bind-pose `sharedMesh` = シェイプキー 0 で固定 | 版固定・来歴(状態非依存・堅い) |
| `radius_m` 等の採寸値 | `BakeMesh` = 実状態(シェイプキー適用後) | 実効寸法 |

自分用なら無害だが、**持ち寄った瞬間**、あるユーザーのバスト縮小状態の値を、別ユーザーの非縮小状態の値と、同じ `meshHash` の下で同一視してしまう。これが [MEASUREMENT_SPEC.md](MEASUREMENT_SPEC.md) §6 で「保留」とした*条件の完全化*の核心 ── 「shapekey 0 で測る前提だが強制していない」不整合の正体である。

## 3. 条件スナップショットを採寸の一級出力にする

解は、計測条件を採寸の**第一級フィールド**に昇格させること:

- **各採寸行に condition snapshot を必須化**: シェイプキー値ベクトル(または順序安定なハッシュ)・**active レンダラー集合**・pose・scale 正規化・tool/threshold バージョン。[MEASUREMENT_SPEC.md](MEASUREMENT_SPEC.md) §6 の `{ meshHash, shapekeyValues, pose, scaleNorm, … }` を「記録すべき」から「無いと採寸を発行しない」へ。
- **active 集合まで含めるのが要点**: シェイプキーでバストを縮めるワークフローなら条件は shapekey 値で足りるが、「胸メッシュを小さい方に差し替える(active 切替)」ワークフローでは、条件は値を超えて**どのメッシュが active か**になる。スキーマはトポロジ選択まで覆う必要がある。
- **比較は同条件同士、または条件差を明示補正でのみ**。条件不一致のまま比べた結果は「誤差あり」とフラグする。これは [SHARING_PROTOCOL.md](SHARING_PROTOCOL.md) の self-consistency(誤り検出器＝f の反例発見器)と同型 ── 条件 mismatch は協調台帳が自分で検出できる種類の誤りになる。
- **共有コーパスは BASE 条件で正規化**(全シェイプキー 0・配布時の active 構成 ＝ アバター配布時の素の状態)。各ユーザーの実効値は BASE からの**個体差分**として表す。これは「誤差」ではなく**カスタマイズ差**であり、区別が立つことが重要。

## 4. 条件空間の連続性 ── 組み合わせ爆発とその畳み方

ここが 3D 特有で、人間服飾と決定的に違う(ユーザー提案, 2026-06-27)。

人間のアパレルでは**体型は固定された離散個体**、服は S/M/L の**離散サイズ**で、マッチングは有限の表に収まる。ところが 3D アバターでは:

- **素体側が連続変形できる** ── シェイプキーは連続値、スケールも連続。
- **衣装側も**サイズ違いメッシュの選択 ＋ 追従(Blendshape Sync)/非追従 の別。

したがって組み合わせは `(avatar, shapekey ベクトル, scale) × (garment, size-variant, follow-mode)` の積となり、離散表では**列挙できない**。素朴には爆発する。

**しかし連続性は呪いであると同時に恵みでもある。** 連続だからこそ畳める:

**(a) 関数化 ── 表でなく場で持つ.** 同一トポロジ上でシェイプキーを動かしたときの周径は、多くの計測点で滑らか・単調・準線形に振る舞う(バスト 0%→100% で胸囲は連続に増える)。ゆえに全点を測って表に持つのでなく、**少数サンプル＋補間で `measurement(shapekey)` という関数(場)として持てる**。これは [CORRECTABILITY_FIELD.md](CORRECTABILITY_FIELD.md) が f を「場」として持つのと同型の発想を、採寸側に適用したもの。

**(b) 1点評価への畳み込み.** ユーザーが実際に知りたいのは常に一つ ── 「**私のこの設定**(素体＋shapekey＋scale)に、**この衣装**は合うか」。全組み合わせ表は要らず、必要なのは「自分の実効採寸(1ベクトル、§2.1 ゆえ mismatch ゼロ)× 衣装の仕上がり寸法(1ベクトル、[MEASUREMENT_SPEC.md](MEASUREMENT_SPEC.md) §4)」のクリアランス比較だけ。**組み合わせ空間は大きくても、評価は常に 1×1 に畳まれる。** そして declipper の貫通修正自体が、その 1 点を実物で解く装置である。

**(c) 予測/門番の二層.** マッチング表(予測層)は粗くてよい。最終判定は declipper の実計算(門番)が出す([FAMILY_MODEL.md](FAMILY_MODEL.md) §9「ラベルは予測、診断が判定」)。だから予測層は連続空間を粗いサンプルで覆えば足り、精度の負担は門番が引き受ける。

**追従衣装は次元を畳む.** Blendshape Sync で素体に追従する衣装は、素体を変形させても一緒に変形して貫通が消える([DETECTION_SEMANTICS.md](DETECTION_SEMANTICS.md))。つまり追従衣装は組み合わせ空間から自動的に脱落し、**組み合わせが本当に効くのは「非追従衣装 × 素体変形」の象限**に限られる ── これは偶然にも declipper の主戦場そのもの。見かけの爆発の多くは Sync が吸収してくれる。

**No Cache との緊張(正直に).** `measurement(shapekey)` 曲線を密に出すと、その素体の「変形の応答=個性」を記述してしまう。出すのはスカラーであって頂点ではない([DESIGN.md](DESIGN.md) §4)が、シェイプキー軸の密な感度関数は形状の癖の記述に近づく。ゆえに共有粒度は **BASE 1 点＋粗い感度**に留め、それ以上の密度は作者許諾の対象とする(公開可否は改変可否と別軸 ── 私的検証は可、公開はスクリーンショット等の条件確認)。

## 5. 含意 ── マッチングは (avatar, user-config) × garment

「バストを縮めてから着る」という習慣は、**同じアバターでも着る人の設定次第で必要な衣装サイズが変わる**ことの実証である。ゆえに [MEASUREMENT_SPEC.md](MEASUREMENT_SPEC.md) §7 のマッチング行列は「アバター BASE × 衣装」では閉じず、`(avatar, user-config) × garment` へ次元が増える。BASE 採寸はあくまで**出発点**で、最終的に意味を持つのは「そのユーザーの実効状態での適合」。

[ECOSYSTEM_VISION.md](ECOSYSTEM_VISION.md) のレモン市場解消(採寸表＝適合情報の開示)の観点でも、開示すべき終点は実効状態の適合であり、BASE 開示はその第一歩にすぎない。§4(a) の感度関数があれば、「BASE ではこう、あなたの設定だとこうずれる」という形で、開示を一点でなく**近傍ごと**に与えられる。

## 6. 正直な留保と実装の前段

- **採寸を走らせる場所が条件になる**: 採寸ダンプを**エディタで**(ビルド前に)走らせると、その瞬間のシーンの active 構成が記録される。検証のために胸メッシュを active に戻して測れば、アップロード構成と違う条件の採寸になる(これも mismatch)。共有採寸は**ビルドパス側で取る**か、**エディタ採寸時の active 構成を条件に刻む**かのどちらかが要る。
- **感度関数のサンプル設計は未定**: §4(a) の `measurement(shapekey)` を何点で測り、どこまで線形と仮定してよいかは未検討(デフォルメ素体は非線形・複数ピークを持ちうる、[MEASUREMENT_SPEC.md](MEASUREMENT_SPEC.md) §2 の極値検出と同じ注意)。較正後に決める。
- **実装の前段**は [MEASUREMENT_SPEC.md](MEASUREMENT_SPEC.md) §6 の*条件の完全化*(シェイプキー状態・active 集合・scale・版を条件に載せる、または計測時に BASE を強制)。本書は概念モデルで、拘束するのは DESIGN/ROADMAP。最良のテストケースは「バストをシェイプキーで縮める＋胸メッシュを active 切替で差し替える」運用 ── 条件スキーマがこの二つを正しく区別できれば、モデルは実地で立つ。

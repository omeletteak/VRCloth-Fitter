# E2E 目視テスト手順

実アバター+衣装で貫通修正パイプラインを通し確認するための手順書です。フェーズ1(ROADMAP)の完成判定に使います。自動テスト(EditMode)の実行方法は [CLAUDE.md](../CLAUDE.md) を、サポート範囲の定義は [DESIGN.md](DESIGN.md) 第9節を参照。最終更新: 2026-06-15。

## 0. 前提と事前チェック

- Unity 2022.3.22f1 でこのプロジェクトを開く(別プロジェクトでテストする場合は [DESIGN.md](DESIGN.md) 第7節のジャンクション方式)
- テスト素材: 素体メッシュを共有し部分的に体型が違うアバターのペア(こまどの新しめアバター群を想定)+ どちらか一方に対応した衣装

実行前のチェックリスト:

- [ ] **素体メッシュが削除されていない** — 普段の運用で素体を消している場合は戻す。消した領域には貫通する相手がいないため、テストの信号が出ない
- [ ] **シェイプキーはショップ既定のまま触らない** — 素体と衣装はショップが詰めた組み合わせで形が合っている。一括ゼロにすると素体形状が変わり偽貫通が出る(素体側 vs 衣装側の扱いと判定方法は §0.1)
- [ ] **アバターが Humanoid** — カプセル生成は Humanoid ボーン前提(`Animator.isHuman`)
- [ ] **体型差の出どころを確認** — 両アバターの対応ボーンのスケールをインスペクタで見比べる。差がボーンスケールに乗っている場合、Merge Armature が衣装ごとスケールして吸収するため貫通がほぼ出ない。**メッシュに焼き込まれた差**(同トポロジで頂点位置が違う)のあるペアを選ぶ

### 0.1 シェイプキーと Blendshape Sync(重要)

シェイプキーは**素体側と衣装側で扱いが違う**。一括ゼロにしてはいけない。

- **素体側** — ショップ既定の値のまま**触らない**。素体メッシュはこの状態のまま SDF コライダ(またはカプセル元)に焼かれる。ゼロに寄せると素体形状が変わり、衣装が想定していない素体に対して**偽の貫通**が出る。特に Sync 対の無い「素体縮小/クリッピング対策」系をオフにすると、素体が隠れない**肌見せ境界=襟元・胸元・ソックス上端**ではみ出す
- **衣装側(Sync駆動)** — Modular Avatar の **Blendshape Sync** が付くシェイプキーは、エディタ上で常時、素体側の値を衣装側へコピーする。よって**単独でゼロにできない**(仕様)。値は素体側で操る
- **衣装側(非Sync)** — Sync の付かない独立シェイプキー(衣装の一部を縮める/隠す shrink・hide 系など)は単独で操作でき、**テスト時は中立(0)にする**。縮め/隠しシェイプキーは畳んだ頂点を**体内へ押し込む**ため、SDF が深い**偽貫通**として拾い偽RED→当該レンダラーごと skip になる(実機で確認: 左ソックス短縮SKで左ソックスが約53mm内側と誤検出 → 0にして解消)。実使用で意図的に shrink ON のまま着る運用への堅牢化はフェーズ3課題

そのためヌルテストの**合否は検出で判定する**:

- **見るもの = Preflight 判定 + シーンビューのヒートマップ**(検出が正しいか)
- **適用後メッシュは合否に使わない** — 衣装側シェイプキーが非0だと、書き戻しが頂点に焼いた変位とランタイム再適用とで**二重適用**され崩れる(フェーズ3のシェイプキー対応までの既知制約)

適用の**見た目**を確認したいときだけ、別途:

- 素体側シェイプキーを **0** にする(Sync で衣装側も 0 に揃う)→ Run。二重適用が消え純粋なフィット結果が見える(テストする体型差が**ベースメッシュに焼かれている**衣装ペアに限る。上のチェックリスト参照)
- 最終状態(Merge Armature 適用後=アップロード相当)を見るには **アバター右クリック ▸ Modular Avatar ▸ Manual Bake Avatar** で複製を生成する。確認後に削除する使い捨て。**二重適用は Manual Bake にも出る**(シェイプキーが非0なら崩れて見える)

## 1. セットアップ

1. アバターをシーンに置く
2. 衣装を Modular Avatar の **Setup Outfit** で着せる(衣装がアバターの子になる)
3. 衣装ルートに **VRClothFitter** コンポーネントを付与 — Target Avatar は親をさかのぼって自動検出される(されない場合は手動で指定)
4. インスペクタで **Show Scene Gizmos** を ON にし、**Preview Body Proxy** を押す
5. **カプセルが体に沿っているかを最初に目視する。** カプセルがズレている・太すぎる・細すぎるなら、以降の検出結果は信用できない(半径ハードコードの既知の限界 — 観察結果はフェーズ3「半径自動推定」の材料として控える)

## 2. 手順0: ヌルテスト(対照実験)

衣装を**本来の対応アバター**に着せて Run Fitting する。30秒でできるカプセル品質の対照実験。

- 期待値: Preflight ログが **GREEN**、`penetrating ≈ 0`

```
[VRClothFitter] Preflight <renderer名>: GREEN — penetrating 0/8000 verts (0.0%),
max 0.0 mm below surface (0% of capsule radius), p95 0.0 mm, largest patch 0.0%, margin-zone hits 96.
```

- `margin-zone hits` が多いのは正常(マージン帯のかすりは侵入に数えない設計)
- ここで黄・赤になるならカプセルの誤検出。その時の数値(max depth・侵入率)としきい値較正のためログを控える

### 手順0b: メッシュSDF コライダで再ヌルテスト(今回の検証ゲート)

カプセルは円断面で胴・足を表せず、ヌルテストでも偽貫通が出る(2026-06-15 時点で胴・足は赤)。これを解消するために導入したのが **メッシュSDF コライダ**([DESIGN.md](DESIGN.md) §6)。同じ対応アバターで切替えて比較する。

1. インスペクタの **Use Mesh SDF Collider** を ON にする(既定 OFF)。素体メッシュは **Body Mesh** 欄で指定可、空なら自動検出
2. もう一度 **Run Fitting**。ログに `... mesh backend` と `Mesh-SDF collider built from '<素体名>' (N verts, M tris)` が出る
3. 期待値: カプセルで赤・黄だった胴・足の偽貫通が **GREEN に近づく**(`penetrating` が大きく減る)。これが本実装の合否信号
4. 確認ポイントと記録:
   - カプセル backend と mesh backend の Preflight 数値を**並べて控える**(偽貫通がどれだけ減ったか)
   - 素体メッシュが大きいと初回 Run は重い(ブルートフォース O(頂点×三角形))。**所要時間も控える** — BVH 高速化(ROADMAP フェーズ1)の要否判断に使う
   - mesh backend で `max ... of capsule radius` の `%` 表記は公称厚みベース(カプセル半径ではない)。判定は絶対深さ主体
- それでも残る偽貫通(素体メッシュの穴・裏面・別パーツの干渉など)があれば箇所と数値を控える — 非水密対策は巻き数判定で入っているが、実機の崩れは较正材料になる

## 3. 手順1: クロス着せ替え(本番)

同じ衣装をもう一方のアバターに着せ替えて実行する。

1. Run 前にシーンビューのヒートマップで貫通箇所を目視(どこが光るか)
2. **Run Fitting**
3. ログの確認ポイント:
   - `Preflight ...: GREEN/YELLOW` と各数値(こまどペアなら緑〜黄の想定)
   - `Push-out + smoothing finished after N pass(es); M vertices still penetrating` — **M = 0** が期待値
   - `Applied fitted mesh copies to N renderer(s)`
4. 目視評価 — 要所は**肌見せ境界×関節**:
   - 袖口⇔肘、襟元・鎖骨、裾⇔太もも、ソックス上端⇔膝
   - 貫通が消えたか
   - 修正領域の境界に膨らみ・段差が見えないか
   - テクスチャの伸びがないか
   - しわ・プリーツ等の衣装ディテールが潰れていないか(変位場方式で保持されるはず)
   - ※適用後メッシュを見る評価なので、§0.1 に従い**素体シェイプキーを 0 にして二重適用を避けた状態**で行う
5. **Before/After のスクリーンショット**を撮る(受け入れ基準)

## 4. 非破壊と Undo の確認

- **Ctrl+Z** で元のメッシュ参照に戻ること(Run 1回分の適用は一手で戻るはず。分かれる場合は数回)
- Project ビューで元の衣装メッシュアセット(FBX)が**未変更**であること
- Undo 後にもう一度 Run すれば再適用できること
- メッシュ複製はシーン内のみでアセットは作られない。シーンを保存する前に Undo で戻せばシーンも汚れない

## 5. ポーズ依存の貫通を見る場合(任意)

> **注意**: エディタ上では Modular Avatar のマージは未実行(ビルド時処理)のため、素体と衣装のアーマチュアは**別物**。素体側のボーンだけ曲げると衣装が追従せず、**偽の貫通**を作ってしまう。

- 正しくは**対応するボーンを両方**同じだけ回す(例: 肘なら素体の LowerArm と衣装側アーマチュアの対応ボーン)
- その状態で Run すればそのポーズでの貫通が修正される。ただし結果は**そのポーズに最適化**される(複数ポーズの束ねはフェーズ3「代表ポーズ対応」)
- 関節の曲げ起因の貫通は体型差ゼロでも出る(素体と衣装のウェイト差×回転角)。これはフェーズ1の合否対象外として観察記録のみ

## 6. 赤(RED)判定が出たら

- それは「この体型差は貫通修正の枠外(リターゲティング級)」というシグナル([DESIGN.md](DESIGN.md) 第9節)。適用は自動でスキップされる
- 枠外の挙動を意図的に見たい場合のみ、インスペクタの **Force Apply (Out of Range)** を ON(結果は崩れる前提。確認後は OFF に戻す)
- しきい値の較正: 「直せてほしいのに赤」「明らかに無理筋なのに緑」と感じたケースは、その時の Preflight 数値を必ず控える — `PreflightDiagnostic` の定数調整の材料になる

## 7. 結果の記録と完了判定

- スクリーンショット+Preflight ログの数値を控え、次の AI セッションに共有する
- 見つかった問題(カプセル不適合、平滑化の過不足など)は **ROADMAP のフェーズ3に追記**する(`bd create` は分裂不具合のため使用不可。`bd close` / `bd update` は使える)
- 完了判定(j3l のクローズ条件):
  - [ ] ヌルテストが GREEN(**メッシュSDF コライダで**。カプセルは胴・足で赤が既知のため、合否はメッシュSDF側で判定。手順0b)
  - [ ] メッシュSDF の所要時間を記録し、BVH 高速化の要否を判断
  - [ ] クロス着せ替えで貫通が解消し、Before/After スクショで確認できる
  - [ ] 新たな破綻(膨らみ・伸び・ディテール潰れ)がない
  - [ ] Undo で完全に元へ戻る

## 8. トラブルシューティング

| 症状 | 確認すること |
|---|---|
| `No active SkinnedMeshRenderer found` | 衣装ルート配下にアクティブな SkinnedMeshRenderer があるか。コンポーネントを付けた場所が衣装ルートか |
| Run Fitting ボタンが押せない | Target Avatar が検出されているか(衣装がアバターの子になっているか) |
| ギズモが見えない | Show Scene Gizmos が ON か。シーンビューを見ているか |
| 対応衣装なのに大量検出 | カプセルが太すぎる(Preview Body Proxy で目視)。margin を下げて切り分け |
| Run しても何も変わらない | 検出 0 件なら正常(ヌルテストとして成功)。ログの Detected 行を確認 |
| 適用後に全体が二重に膨らんで崩れる | 衣装側シェイプキーが非0(Blendshape Sync で素体に追従)。検出は正常で、適用の二重適用による既知制約。適用の見た目を見るなら素体シェイプキーを 0 にしてから Run(§0.1) |

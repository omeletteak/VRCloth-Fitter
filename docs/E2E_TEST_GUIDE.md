# E2E 目視テスト手順

実アバター+衣装で貫通修正パイプラインを通し確認するための手順書です。フェーズ1(ROADMAP)の完成判定に使います。自動テスト(EditMode)の実行方法は [CLAUDE.md](../CLAUDE.md) を、サポート範囲の定義は [DESIGN.md](DESIGN.md) 第9節を参照。最終更新: 2026-06-17。

## 0. テストの2段階(最初に読む — 重要)

VRCloth-Declipperの**本番は NDMFパス(Merge Armature 後・ビルド時)に走る**想定です(パス順序スパイク 2026-06-14 で確認済み)。一方、インスペクタの Run Fitting を**編集モードで押すとマージ前**の状態を見ます。両者は別物です:

- 編集モードでは衣装ボーンが素体にマージされておらず、**Merge Armature がボーンスケールで吸収するはずの体型差が未反映**(プレイモードON/Bakeで反映され、OFFで戻る)
- そのため編集モードで見える貫通には、**マージ後に消える「偽の貫通」が混じる**(特にボーンスケール差の大きいペア)

よってテストは2段階に分けます。**各手順に【段階1】【段階2】を明記**します。

| | 段階1: 編集モード(マージ前) | 段階2: Manual Bake 後(マージ後・本番相当) |
|---|---|---|
| 見るもの | カプセル品質・検出ロジック・操作の素早い確認 | 貫通解消・ソルバ比較・受け入れの**最終判定** |
| 状態 | Setup Outfit 済み・未マージ | Manual Bake でマージ済み(アップロード相当) |
| 注意 | ボーンスケール差由来の偽貫通が混じる。**最終判定には使わない** | 使い捨て複製。非破壊/Undo の確認は段階1の元シーンで |

## 1. 前提と事前チェック

- Unity 2022.3.22f1 でこのプロジェクトを開く(別プロジェクトでテストする場合は [DESIGN.md](DESIGN.md) 第7節のジャンクション方式)
- テスト素材: **体型差がメッシュに焼き込まれたアバターのペア**(ボーンスケール差は Merge Armature が吸収するため不可)+ **その体型差に自動追従しない衣装**。素材選びには落とし穴があるため、選定ルールと適性確認は §1.2 にまとめた

実行前のチェックリスト:

- [ ] **素体メッシュが削除されていない** — 普段の運用で素体を消している場合は戻す。消した領域には貫通する相手がいないため、テストの信号が出ない
- [ ] **シェイプキーはショップ既定のまま触らない** — 素体と衣装はショップが詰めた組み合わせで形が合っている。一括ゼロにすると素体形状が変わり偽貫通が出る(§1.1)。**特に「素体が衣装からはみ出す」のを見て素体側シェイプキーで縮めてはいけない — それはこのツールが直す貫通そのもので、縮めるとテストが無効になる**(実機で確認: 素体を縮めると初期検出数が変わり比較不能)
- [ ] **アバターが Humanoid** — カプセル生成は Humanoid ボーン前提(`Animator.isHuman`)
- [ ] **体型差の出どころを確認(§4-B で Bake 切り分け)** — 差がボーンスケールに乗っている場合、Merge Armature が衣装ごとスケールして吸収するため**マージ後は貫通がほぼ出ない**。**メッシュに焼き込まれた差**(同トポロジで頂点位置が違う)のあるペアでないとクロステストにならない。編集モードの見た目では判別できないので、段階2-B で Bake して切り分ける

### 1.1 シェイプキーと Blendshape Sync(重要)

シェイプキーは**素体側と衣装側で扱いが違う**。一括ゼロにしてはいけない。

- **素体側** — ショップ既定の値のまま**触らない**。素体メッシュはこの状態のまま SDF コライダ(またはカプセル元)に焼かれる。ゼロに寄せると素体形状が変わり、衣装が想定していない素体に対して**偽の貫通**が出る。特に Sync 対の無い「素体縮小/クリッピング対策」系をオフにすると、素体が隠れない**肌見せ境界=襟元・胸元・ソックス上端**ではみ出す
- **衣装側(Sync駆動)** — Modular Avatar の **Blendshape Sync** が付くシェイプキーは、エディタ上で常時、素体側の値を衣装側へコピーする。よって**単独でゼロにできない**(仕様)。値は素体側で操る
- **衣装側(非Sync)** — Sync の付かない独立シェイプキー(衣装の一部を縮める/隠す shrink・hide 系など)は単独で操作でき、**テスト時は中立(0)にする**。縮め/隠しシェイプキーは畳んだ頂点を**体内へ押し込む**ため、SDF が深い**偽貫通**として拾い偽RED→当該レンダラーごと skip になる(実機で確認: 左ソックス短縮SKで左ソックスが約53mm内側と誤検出 → 0にして解消)。実使用で意図的に shrink ON のまま着る運用への堅牢化はフェーズ3課題

### 1.2 クロステスト素材の選び方(自動追従の罠)

クロステストは「**体型差で残る貫通を Run が直せるか**」を見る。成立条件は、貫通が **Run 以外の要因で消えないこと**。消す要因は2つあり、どちらも素材選びで避ける:

1. **ボーンスケール差** — Merge Armature が衣装ごとスケールして吸収し、マージ後に貫通が消える → **体型差がメッシュに焼き込まれた**(同トポロジで頂点位置が違う)ペアを選ぶ(判別は §4-B の Bake 切り分け)
2. **衣装の体型自動追従** — 素体の体型シェイプキーを衣装へコピーする衣装(Blendshape Sync 内蔵)は、素体を変えた**瞬間に衣装も追従変形して貫通が消える**。Run の効果と切り分けられず、テストにならない

衣装は体型適応の仕方で2種に分かれ、(2)の罠の有無が決まる:

- **固定形状(自動追従なし)** — 特定素体に合わせて形が作り込まれ、体型シェイプキーへの Sync を持たない(アバター付属の純正衣装に多い型)。別体型の素体に着せると体型差ぶんだけ素直に貫通し、**Run だけがそれを直す → クロステストに最適**
- **多素体対応(自動追従あり)** — 多くのアバターに着られるよう体型シェイプキーへの Blendshape Sync を内蔵する(汎用衣装に多い型)。対応素体に着せると**貫通が自前で消える**ため、同一素体系では Run の効果を覆い隠す

**選定ルール**:

- **同一素体系**(姉妹アバター等・リグ互換で着せ替えが容易)で組むなら、衣装は**固定形状(自動追従なし)**を選ぶ。多素体対応衣装は追従して罠になる
- **別素体系**(作家の違うアバター)で組むなら、多素体対応衣装でも Sync の binding 名が相手素体と一致せず**不発になりやすく**罠を避けられる。ただし体型差が大きすぎると RED→skip になりうる
- いずれも素体は **§1.1 の通りショップ既定のまま**(シェイプキーをいじらない)

**適性の最終確認(各5秒)**:

1. **自動追従テスト** — 素体の体型シェイプキーを少し動かす。**衣装も変形したら自動追従あり**(同一素体系では不適。別衣装に替えるか別素体系で組む)。動かなければ固定形状で適する
2. **焼き込みテスト** — Manual Bake 後も貫通が残れば焼き込み差で有効(§4-B)。消えるならボーンスケール差で不適

## 2. セットアップ

1. アバターをシーンに置く
2. 衣装を Modular Avatar の **Setup Outfit** で着せる(衣装がアバターの子になる)
3. 衣装ルートに **VRClothDeclipper** コンポーネントを付与 — Target Avatar は親をさかのぼって自動検出される(されない場合は手動で指定)
4. インスペクタで **Show Scene Gizmos** を ON にし、**Preview Body Proxy** を押す
5. **カプセルが体に沿っているかを最初に目視する。** カプセルがズレている・太すぎる・細すぎるなら、以降の検出結果は信用できない(半径ハードコードの既知の限界 — 観察結果はフェーズ3「半径自動推定」の材料として控える)

## 3. 段階1: 編集モードの素早い確認(マージ前)

> ここで見える貫通は**偽貫通が混じる**(マージ前)。カプセル品質・検出ロジック・操作の確認に使い、**貫通解消やソルバの最終判定には使わない**(それは段階2)。

### 手順0: ヌルテスト(対照実験)

衣装を**本来の対応アバター**に着せて Run Fitting する。30秒でできるカプセル品質の対照実験。

- 期待値: Preflight ログが **GREEN**、`penetrating ≈ 0`

```
[VRClothDeclipper] Preflight <renderer名>: GREEN — penetrating 0/8000 verts (0.0%),
max 0.0 mm below surface (0% of capsule radius), p95 0.0 mm, largest patch 0.0%, margin-zone hits 96.
```

- `margin-zone hits` が多いのは正常(マージン帯のかすりは侵入に数えない設計)
- ここで黄・赤になるならカプセルの誤検出。その時の数値(max depth・侵入率)としきい値較正のためログを控える

### 手順0b: メッシュSDF コライダで再ヌルテスト

カプセルは円断面で胴・足を表せず、ヌルテストでも偽貫通が出る(2026-06-15 時点で胴・足は赤)。これを解消するために導入したのが **メッシュSDF コライダ**([DESIGN.md](DESIGN.md) §6)。同じ対応アバターで切替えて比較する。

1. インスペクタの **Use Mesh SDF Collider** を ON にする(既定 OFF)。素体メッシュは **Body Mesh** 欄で指定可、空なら自動検出
2. もう一度 **Run Fitting**。ログに `... mesh backend` と `Mesh-SDF collider built from '<素体名>' (N verts, M tris)` が出る
3. 期待値: カプセルで赤・黄だった胴・足の偽貫通が **GREEN に近づく**(`penetrating` が大きく減る)。これがメッシュSDFバックエンドの合否信号
4. 確認ポイント: カプセル backend と mesh backend の Preflight 数値を並べて控える。残る偽貫通(素体メッシュの穴・裏面・別パーツ干渉)があれば箇所と数値を控える

## 4. 段階2: Manual Bake 後の検証(マージ後・本番相当)

ここが**最終判定**です。`Manual Bake Avatar` でマージ後の状態を作り、本番(NDMFパス)に近い条件でクロステストとソルバ比較を行います。

> **なぜ Bake が要るか**: 編集モードはマージ前で、ボーンスケール差が未吸収。本番は Merge Armature 後に走るので、テストもマージ後で見る。NDMFパス化(フェーズ5)が入れば自動化されるが、現状は手動 Bake で代替する。

### 手順A: マージ後の複製を作る

1. アバター右クリック ▸ **Modular Avatar ▸ Manual Bake Avatar**。マージ済みの複製が生成され、元アバターは非アクティブ化される
2. 複製は**使い捨て**。確認後に削除する。シーンは保存しない(段階1の非破壊な元シーンを汚さない)

### 手順B: 体型差の切り分け(このペアがクロステストに使えるか)

Bake 複製で、編集モードの段階1で見えていた貫通が**マージ後も残っているか**を確認する。

- 衣装の SkinnedMeshRenderer を選び、シーンビューで素体との関係を目視(または手順C で Preflight を見る)
- **マージ後も貫通が残る** → メッシュに焼き込まれた体型差。VRCloth-Declipperが直すべき**本物**。手順C へ進む
- **マージ後に貫通が消えた/激減した** → ボーンスケール差で吸収された偽貫通。**このペアはクロステストに不適**。§1 の「メッシュに焼き込まれた差」のあるペアを選び直す

### 手順C: クロステスト本番(Bake 複製で Run)

> **注意(IEditorOnly)**: `VRClothDeclipper` は `IEditorOnly` のため Manual Bake で**削除される**。Bake 複製の衣装には **VRClothDeclipper を付け直す**必要がある(これがフェーズ5 NDMFパス化=Bake工程内で自動実行、が本来の解である理由)。

1. Bake 複製の衣装ルートに **VRClothDeclipper を付け直す**(Target は複製アバターを指す)。Use Mesh SDF Collider = ON
2. Run 前にシーンビューのヒートマップで貫通箇所を目視(どこが光るか)
3. **Run Fitting**
4. ログの確認:
   - `Preflight ...: GREEN/YELLOW` と各数値
   - `Push-out + smoothing finished after N pass(es); M vertices still penetrating` — **M = 0** が期待値
   - `Applied fitted mesh copies to N renderer(s)`
5. 目視評価 — 要所は**肌見せ境界×関節**(袖口⇔肘、襟元・鎖骨、裾⇔太もも、ソックス上端⇔膝):
   - 貫通が消えたか / 修正領域の境界に膨らみ・段差がないか / テクスチャの伸びがないか / しわ・プリーツ等のディテールが潰れていないか
6. **Before/After のスクリーンショット**を撮る(受け入れ基準)
7. RED が出たら §6 へ

### 手順D: ソルバ比較(coarse vs projected)

現行ソルバ(**coarse**)と試作の法線/接線分解ソルバ(**projected**)を比較する。projected は平滑化1ステップごとに侵入頂点をマージン面へ再投影して再沈み込みを防ぐ設計([DEFORMATION_METHODS.md](DEFORMATION_METHODS.md) §3.1)。既定 **OFF(coarse)**。

> Bake 複製は毎回作り直せばクリーンな初期状態になる(段階1の「シーン再オープン」に相当)。これを使って同一条件で比較する。

1. Bake 複製を作り、VRClothDeclipper を付け直し、**Use Projected Solver = OFF**(coarse)で Run。ログの **`Detected N penetrating vertices`(初期検出数 N)**、`finished after P pass(es); M vertices still penetrating` の **M**、所要時間(体感)を控える。スクショ
2. Bake 複製を**作り直す**(または別の Bake 複製で)。**Use Projected Solver = ON**(projected)で Run
3. **公平性チェック(必須)**: projected 側の **初期検出数 `Detected …` が手順1の N と一致**することを最初に確認。一致しなければ初期状態が違う(別 Bake/別状態)ので作り直してやり直す。一致して初めて比較成立
4. 比較の観点(**パス数は直接比較しない** — coarse は収束で決まる適応値〈最大 `maxPasses`〉、projected は固定反復数〈`iterations`〉で意味が違う。残留貫通=0 を両者が満たした上で質で比べる):
   - **残留貫通(M)** — どちらも 0 が期待。projected で 0 にならない箇所があれば位置と数値を控える
   - **段差・膨らみ** — 押し出し境界の馴染みが projected で改善するか(本命の差。肌見せ境界×関節を重点)
   - **ディテール保持** — しわ・プリーツが潰れていないか
   - **所要時間** — projected は毎反復で全頂点を再スキャンするため重い(実機: 約2倍の体感)。見た目の改善がそれに見合うか
5. 所見は ROADMAP フェーズ3 / [DEFORMATION_METHODS.md](DEFORMATION_METHODS.md) §3.1 へフィードバック。**projected を既定化するか/`iterations` を露出するか**の判断材料

## 5. 非破壊と Undo の確認(段階1の元シーンで)

> Bake 複製は元から使い捨て。非破壊性は段階1のマージ前シーンで確認する。

- **Ctrl+Z** で元のメッシュ参照に戻ること(Run 1回分の適用は複数レンダラー分=数手に分かれることがある)
- Project ビューで元の衣装メッシュアセット(FBX)が**未変更**であること
- Undo 後にもう一度 Run すれば再適用できること
- メッシュ複製はシーン内のみでアセットは作られない。シーンを保存する前に Undo で戻せばシーンも汚れない

## 6. 赤(RED)判定が出たら

- それは「この体型差は貫通修正の枠外(リターゲティング級)」というシグナル([DESIGN.md](DESIGN.md) 第9節)。適用は自動でスキップされる
- 枠外の挙動を意図的に見たい場合のみ、インスペクタの **Force Apply (Out of Range)** を ON(結果は崩れる前提。確認後は OFF に戻す)
- しきい値の較正: 「直せてほしいのに赤」「明らかに無理筋なのに緑」と感じたケースは、その時の Preflight 数値を必ず控える — `PreflightDiagnostic` の定数調整の材料になる
- **胸など肌見せ部に深い RED が集中**する場合、衣装の布が体型に対して物理的に足りない(覆いきれない)可能性がある。押し出しでは解けず、衣装側のリターゲ/作り直しが要る領域。RED でスキップされるのが正しい

## 7. ポーズ依存の貫通を見る場合(任意)

> **注意**: Bake 前(段階1)は MA マージ未実行のため、素体側のボーンだけ曲げると衣装が追従せず**偽の貫通**を作る。ポーズ確認は Bake 後(マージ済みで単一アーマチュア)で行うのが確実。

- Bake 後ならボーンが統合されているので、関節を曲げると衣装も追従する。その状態で Run すればそのポーズでの貫通が修正される(結果はそのポーズに最適化される)
- 関節の曲げ起因の貫通は体型差ゼロでも出る(素体と衣装のウェイト差×回転角)。これはフェーズ1の合否対象外として観察記録のみ(本命はフェーズ3「代表ポーズ対応」)

## 8. 結果の記録と完了判定

> **合否は可視の外側面で見る** — Preflight 残留は「押し出しの到達度」であって合否そのものではない。厚物衣装(靴・ブーツ等)は内側メッシュが素体に正当接触し残留が0にならないが、**可視の外側面に素体が出ていなければ合格**。残留数と可視合否は直結しない(偽陽性の整理は [DETECTION_SEMANTICS.md](DETECTION_SEMANTICS.md))。

- スクリーンショット+Preflight ログの数値を控え、次の AI セッションに共有する
- ソルバ比較(手順D)を行った場合は coarse / projected の数値(初期検出数・残留貫通・所要時間)と所見も控える
- 見つかった問題(カプセル不適合、平滑化の過不足など)は **ROADMAP のフェーズ3に追記**する(`bd create` は分裂不具合のため使用不可。`bd close` / `bd update` は使える)
- 完了判定(j3l のクローズ条件、**判定はすべて段階2=Bake 後で**):
  - [ ] ヌルテストが GREEN(メッシュSDF コライダで。カプセルは胴・足で赤が既知。段階1 手順0b で確認)
  - [ ] メッシュSDF の所要時間を記録し、BVH 高速化の要否を判断
  - [ ] **Bake 後**でクロス着せ替えの貫通が解消し、Before/After スクショで確認できる(段階2 手順C)
  - [ ] 新たな破綻(膨らみ・伸び・ディテール潰れ)がない
  - [ ] Undo で完全に元へ戻る(段階1 §5)

## 9. トラブルシューティング

| 症状 | 確認すること |
|---|---|
| `No active SkinnedMeshRenderer found` | 衣装ルート配下にアクティブな SkinnedMeshRenderer があるか。コンポーネントを付けた場所が衣装ルートか |
| Run Fitting ボタンが押せない | Target Avatar が検出されているか(衣装がアバターの子になっているか) |
| ギズモが見えない | Show Scene Gizmos が ON か。シーンビューを見ているか |

# VRCloth-Fitter 設計ドキュメント

このドキュメントは、VRCloth-Fitter の設計判断とその背景(競合分析・プライバシー所見)を記録します。最終更新: 2026-06。

## 1. プロジェクトの目的

VRCloth-Fitter の目的は、**衣装フィッティング領域にオープンソースの実装が存在すること**です。収益は目的ではありません。

VRChat の改変エコシステムでは、Modular Avatar・NDMF・Avatar Optimizer といった基盤ツールがオープンソース(MIT)でインフラとして機能しています。衣装フィッティング(体型変換・貫通修正)はいまのところ有料ツールが担っており、ここにも**中身を検証でき、作者の活動が止まっても使い続けられる**オープンな選択肢が一つあってよい — それが本プロジェクトの動機です。オープンであること自体が、後述する No Cache 原則をコードで検証可能にする手段でもあります。

## 2. 解く問題: 「着せた後」の貫通

Modular Avatar の Merge Armature(Setup Outfit)は骨格を揃えてくれますが、素体と衣装の体型の差までは吸収しません。結果として、対応衣装ですら着せた後に素体が衣装を突き抜ける「貫通」が残り、ユーザーはシェイプキー調整・メッシュ削除・Blender での手修正でしのいでいます。

VRCloth-Fitter はここを自動化します。**どの方法で着せたかは問いません** — 対応衣装の手動着せ替えでも、変換ツールが出力した衣装でも、着せ終わった状態に対する後処理として機能します。

どこまでの体型差を「貫通修正」として扱うか(どこからがリターゲティングか)の定量的な線引きは第9節で定義します。

## 3. 競合分析(2026-06 時点)

| | もちふぃった～ | Alterith | VRCloth-Fitter |
|---|---|---|---|
| 価格/ライセンス | ¥2,500・ソース非公開(Blender アドオンは GPL-3.0) | ¥3,000・再配布不可 | 無料・MIT(ソース公開) |
| 担当領域 | 体型変換(リターゲティング) | 体型変換(リターゲティング) | **貫通修正(後処理)** |
| 方式 | 中間素体経由の2段階変換。アバターごとの「変換プロファイル」が必要(主にショップが配布) | ローカル完結。衣装+変換元アバター+変換先アバターの3点をシーンに置く | ローカル完結。着せ終わった状態だけが入力 |
| 貫通への対応 | 変換の範囲外(FAQ で、変換後の貫通は MeshCutter 等の別手段で対応する旨を案内) | フィット時の静的マージン(Minimum Margin 等)で予防。着せた後の検出ベース修正は範囲外 | これが担当領域 |
| 中間データの形態 | アバターごとの変換プロファイル(配布可能なファイル) | なし(シーン内完結) | なし(No Cache) |

**読み取れること:**

- 体型変換は2方式(プロファイル配布型・ローカル完結型)とも先行製品があり、機能面の空白はない
- 一方「着せた後の貫通修正」は両製品とも扱っておらず、もちふぃった～の FAQ は明示的に他の手段へ誘導している。**ここが空白であり、本プロジェクトの担当領域**
- 変換ツールが普及するほど「変換したが貫通する」ケースは増えるため、本プロジェクトは競合ではなく両製品の補完になる

## 4. プライバシー所見: 体型変換は原理的に素体形状を運ぶ

本プロジェクトの No Cache 原則は、特定製品の欠点ではなく、**体型変換という処理が原理的に持つ構造的な性質**から導かれます。

- アバター A の衣装をアバター B の体型へ変形するには、両者の体型の対応(差分・変形場)が要る。この対応データは解像度が上がるほど元アバターの素体形状を高い精度で内包し、究極的には素体メッシュの点群そのものに漸近する
- したがって「体型差分」や「変換プロファイル」を**ファイルとして保存・配布する**形式は、原理的に元アバターの体型情報を持ち運ぶ。安全性は権利者(アバター作者・ショップ)の同意のもとで配布されるという**運用**に依存し、技術だけでは保証されない
- これは特定製品への批判ではなく、設計上の出発点です。**変換は体型情報を運ばざるを得ない**という事実を受け入れた上で、本プロジェクトはそれを運ばずに済む領域(着せた後の貫通修正)に限定し、運用に頼らず技術的に形状を残さない設計を選びました(第5節)

## 5. 設計原則: No Cache

上記の教訓から、VRCloth-Fitter は次を原則とします。

> **アバター素体形状を復元しうるデータを、ディスクに保存せず、ツールの外に出さない。**

- 貫通修正に必要な計算(プロキシ生成・距離場・変位)はすべてエディタセッション内のメモリで行い、その場で適用してその場で捨てる
- 貫通修正は「ユーザーのプロジェクト内に既にあるもの」だけで完結するため、リターゲティングと違って形状データを外部化する必要が**そもそもない**。No Cache はこの性質があって初めて成立する(本プロジェクトが貫通修正に特化する理由のひとつ)
- 差分の保存・共有機能(旧構想)は凍結。安全性を保証できる形式(局所化・量子化・ノイズ付加など)の議論が固まらない限り実装しない
- オープンソース(MIT)であることは、この原則が守られていることを誰でもコードで検証できるようにするための手段でもある

## 6. アーキテクチャ

パイプラインは5段で、すべて Unity エディタ内で完結します。

```
プロキシ生成 → 貫通検出 → 押し出し → ラプラシアン平滑化 → 非破壊適用
```

1. **プロキシ生成**(`VRClothProxyGenerator`) — Humanoid ボーンから `BodyCapsule` 列を生成。素体メッシュの代わりにカプセルを衝突判定に使うことで、計算を軽量に保ち、素体形状そのものへの依存も減らす(この衝突表現の見直しは下記「設計判断: 衝突表現をカプセルからメッシュSDFへ」を参照)
2. **貫通検出**(`VRClothPenetrationDetector`) — 衣装全頂点×全カプセルの符号付き距離を計算し、マージン込みで内側の頂点を `PenetrationHit`(頂点 index・位置・深度・最近接カプセル)としてマーク
3. **押し出し**(`PenetrationPushOut`) — 侵入頂点が表面+マージンに乗るための変位を、カプセル SDF の勾配方向に沿って**変位場**(頂点ごとの変位ベクトル配列)へ記録
4. **ラプラシアン平滑化**(`LaplacianSmoothing`) — 頂点座標ではなく**変位場**に対して、動いた頂点の近傍リング限定で適用し、修正量を滑らかに分布させる。平滑化→再検出→再押し出しを 2〜3 回反復し、最後は押し出しで終える
5. **非破壊適用**(`VRClothMeshApplier`) — 元座標+変位場の合成結果を複製メッシュに書き戻して差し替え。元のメッシュアセットは変更しない

### 設計制約: 平滑化は変位場へ、押し出しは SDF 勾配方向(2026-06-12 決定)

ソルバ(`PenetrationSolver`)は次の3点を制約として実装します。

**(a) ラプラシアン平滑化の適用対象は頂点座標ではなく押し出し変位場**

- 座標への平滑化は、修正領域内の元ジオメトリ(しわ・プリーツ・裾などの衣装ディテール)まで近傍平均へ引き寄せて潰してしまう(Laplacian 収縮)。変位場への平滑化なら元形状はそのまま残り、**修正量だけ**が滑らかに分布する
- 変位がゼロの頂点は厳密に動かない。したがって各頂点の移動量は修正量(侵入深度)のオーダーで上から抑えられ、「直したい所しか触らない」が構造的に保証される
- 領域境界の錨(アンカー)が「境界外は変位 0」として自然に成立し、座標平滑化で必要だった特別な境界処理を概念ごと不要にする

**(b) 押し出し方向はカプセル SDF の勾配方向に統一**

- 勾配は空間の全点で定義される一貫した方向場で、平滑化後・反復中の再押し出しでも同一の規則で変位を合成できる
- 複数カプセル(min-SDF の合成体)でも「最近接カプセルの勾配」として一意に定まる
- カプセル単体では軸上の縮退点を除き従来の最近傍点方向と数学的に一致するため、挙動の連続性は保たれる(縮退点は垂直フォールバック)

**(c) 反復構成は 押し出し →[変位場平滑化 → 再検出 → 再押し出し]× 2〜3 回**

- 平滑化は変位を侵入側へ戻しうるため、必ず再検出と再押し出しが後に続き、**シーケンスの最後は押し出しで終わる**。これにより最終状態の侵入ゼロが構成的に保証される

なお、この平滑化方式の限界と発展候補(変位場の調和補間、XPBD、ARAP、RBF 空間変形場)の比較検討は [DEFORMATION_METHODS.md](DEFORMATION_METHODS.md) にまとめています。採用を決めた手法は本節に昇格させます。

### 設計判断: 衝突表現をカプセルからメッシュSDFへ(2026-06-15)

ボーン由来のカプセル列(上記プロキシ生成)は初期の素体モデルですが、実機 E2E のヌルテスト(対応衣装を本来のアバターに着せる対照実験)で原理的な限界が出ました。**円断面の掃引体であるカプセルは、断面が楕円で胸が前に張り出す胴や、足・手を表現できません。** 半径を素体メッシュから自動推定しても(フェーズ3 v1)断面形状のズレは残り、密着衣装を本人の体に着せても胴・足が偽の貫通として検出されます。

そこで素体表現を、**セッション内で素体メッシュから構築する符号付き距離(SDF)コライダ**へ移します。第9節の天井2で将来候補として挙げていた手であり、素体メッシュはエディタ時にメモリ上で読み・BVH を建て・その場で捨てるため No Cache と両立します(半径推定が既に同じく素体メッシュを読んでいます。保存・外部化しないという保証は不変)。

- パイプライン2〜5段(検出・押し出し・平滑化・適用)は衝突プリミティブに依存せず**そのまま残ります**。差し替えは「符号付き距離」と「押し出し先(表面+マージン)」の2演算を満たす衝突バックエンドの抽象(`IBodyCollider`)に集約されます
- カプセル半径の「素体形状への依存を減らす」という当初の利点は、No Cache の本質(=保存・外部化しないこと)からは独立した弱い論拠でした。メモリ内で素体メッシュを読むこと自体は原則に反しません
- カプセルは高速プレビュー/フォールバックとして残しうる
- **実装 landed(2026-06-15)**: 抽象 `IBodyCollider`(符号付き距離・勾配・局所厚み)と `MeshSdfCollider`・`CapsuleBodyCollider` を実装し、パイプライン2〜5段+プリフライトを抽象越しに通した。符号判定は一般化巻き数の**絶対値** > 0.5(メッシュの大域的な巻き方向に依存せず、穴・自己交差にも堅牢 — §9 が挙げる非水密メッシュの懸念に対応)。距離は最近接三角形点(Ericson)。ユニットテストで立方体メッシュの符号付き距離・勾配・押し出し・穴あきメッシュの内外判定を検証(84 件緑)。**未検証**: 実機メッシュでの偽貫通解消(E2E)と性能 — ブルートフォース O(頂点×三角形) のため BVH 高速化が残課題。既定はカプセル(`useMeshSdfCollider` トグルで切替)で、E2E 較正までこのまま

**コード配置:**

- `Assets/VRCloth-Fitter/Core/` — 純粋な数学・型(`BodyCapsule`、`PenetrationHit`)。asmdef 化済みで EditMode テストの対象
- `Assets/VRCloth-Fitter/Editor/` — パイプライン、インスペクタ、シーンビュー可視化(`VRClothDebugVisualizer`)
- `Assets/VRCloth-Fitter/Runtime/` — アバターに付くコンポーネント(`VRClothFitter` など)
- `Assets/VRCloth-Fitter/Tests/` — EditMode テスト

可視化(カプセルのワイヤ表示・侵入頂点の深度ヒートマップ)もセッション内メモリのみで動作し、何も保存しません。

## 7. 配布形態と開発環境

- 配布単位は `Assets/VRCloth-Fitter/` フォルダ(VPM パッケージ)。**リポジトリ全体を他プロジェクトへコピーしてはいけません**(VRChat SDK が二重になる)
- 配布は標準 VPM リポジトリ形式を予定。日本語圏では VCC 代替の [ALCOM](https://vrc-get.anatawa12.com/ja/alcom/)(vrc-get ベース)が主流だが、ALCOM と VCC は VPM リポジトリ設定を共有するため、同一の配布で両方の利用者に対応できる
- リポジトリは SDK 等の再配布禁止物を含みません(git 履歴からも除去済み)。クローン後は `vrc-get resolve` か ALCOM / VCC で依存を復元
- 別プロジェクトでのテストはジャンクション推奨: `New-Item -ItemType Junction -Path <テスト側>/Assets/VRCloth-Fitter -Target <クローン側>/Assets/VRCloth-Fitter`
- 将来の VPM 配布には Runtime/Editor の asmdef 整備が必要(パッケージ内の asmdef なしコードはコンパイルされないため)。それまでは Assets 内設置のみサポート

## 8. 凍結した構想(旧 FUTURE_PLANS.md)

旧ドキュメントには「体型差分プリセット(アバター A→B の差分データ)の共有」と「プリセットハブ(オンラインデータベース)」の構想がありました。これらは次の理由で凍結します。

- 体型差分の共有という方向は、変換型の先行ツールが既に担っており機能的な空白がない
- 第4節の通り、保存・配布する差分データは原理的に元アバターの体型情報を含み、No Cache 原則と正面から衝突する
- 再開する場合は、形状復元を不可能にする安全化形式の設計と検証を先行させること

ローカル完結型の体型変換(変換元アバターも所持している前提)だけは No Cache と両立するため、将来の拡張候補としてロードマップのフェーズ4に残しています(ただし Alterith が同方式を実現済みのため優先度は低い)。

## 9. サポート範囲: どこまでの体型差を「貫通修正」と呼ぶか(2026-06-12 定義)

一行で: **貫通修正は「半径方向・局所・まだら」な差を直す後処理(深さ cm 級まで)。「接線方向・大域・一様」な差はリターゲティングであり対象外。**

### 3つの天井

サポートできる体型差は、次の独立した3つの上限の最小値で決まります。

1. **幾何学(ひずみ)** — 深さ d の侵入をブレンド半径 r で馴染ませると、布の面内ひずみは ε ≈ (d/r)²/2。UV は変わらないため幾何の伸びはそのままテクスチャの伸びとして見える。目につき始める ε≈3% から逆算すると **d/r ≈ 1/4**、つまり見られる品質には深さの4倍のブレンド半径が必要。r はシルエットが許す範囲で頭打ちになるため、d の実用限界は cm 級。これは平滑化を高級化(調和補間・XPBD 等、[DEFORMATION_METHODS.md](DEFORMATION_METHODS.md))しても消えない幾何定数(XPBD は接線方向への逃がしでドレープ系に限り 2〜3 倍稼げる)
2. **プロキシ(素体表現の誤差)** — 素体をカプセルで近似すると、円断面では表せない胴・足で、修正の深さによらず**ヌルテスト段階から**誤差が露出する(実機 E2E 確認)。緩和は半径自動推定では足りず、**セッション内で素体メッシュから構築する SDF コライダ**へ移す(第6節「設計判断」、保存しないので No Cache と両立)。この天井は実質ほぼ解消する見込み(フェーズ1スパイクで検証)
3. **関係性(操作の方向)** — 押し出しは法線(SDF 勾配)方向の操作であり、直せるのは「服はその場所にあるべきで、局所的にめり込んでいるだけ」のケースのみ。裾の高さ・ネックラインの位置・スカートの開きが変わる差は布の接線方向の輸送(再ドレープ)が必要で、**深さによらず**枠外。衣装ベンダーが「胸大用」シェイプキーを同梱する慣行は、この種の差は作り込みで吸収すべきもの(自動修正の枠外)という生態系側の答えでもある

### 枠の形

- **片側性** — 扱うのは「体が服より局所的に大きい」方向のみ。逆方向(体が細い)は貫通自体が起きない — 壊れていない、ただ緩いだけ
- **面積でも破綻する** — 深さが浅くても、衣装頂点の3割が侵入しているならサイズ違いの服=リターゲティング案件

### 判定の目安(初期値)

| 判定 | 目安 | 扱い |
|---|---|---|
| 緑 | 深さ ≤ 1cm 級(局所カプセル半径の〜15%)かつ侵入率 ≤ 10% | 設計どおり。同一素体の派生アバター・ウェイト差由来など |
| 黄 | 深さ 1〜3cm | 条件付きベストエフォート。膨らみ・テクスチャ伸び・カプセル形状の露出が出始める |
| 赤 | 深さ > 3cm、侵入率 > 30%、または接線再配置が必要な差 | 対象外(リターゲティング)。警告して適用を既定で中止 |

この線引きは口約束にせず、**プリフライト診断**として実装しています(`PreflightDiagnostic`)— Run 時の検出統計(最大深度・p95 深度・侵入頂点率・最大連結パッチ面積比・深さ/カプセル半径比)から緑/黄/赤を判定し、赤は適用を既定で中止する(明示オプションでのみ強行可)。約束をコードで検証可能にする点で、No Cache を MIT 公開で担保するのと同じ流儀です。しきい値は上表を初期値とし、実機 E2E(こまど素体共有ペア)で較正します。受け入れの目安は「対応衣装のヌルテストで緑、こまどクロス着せ替えで緑〜黄、明らかなサイズ違いの衣装で赤+適用中止」。

なお判定指標のうち「深さ/カプセル半径比」はカプセル固有で、素体表現をメッシュSDFへ移す際は絶対深さ+局所厚みへ置き換え、しきい値もそこで再較正します。**2026-06-15 時点でカプセルのヌルテストは胴・足で赤** — これが第6節の素体表現見直しの引き金になりました。

なお、この診断を個人ツールの安全装置に留めず、購入判断やサイズ設計に使える互換性標準へ育てる構想を [ECOSYSTEM_VISION.md](ECOSYSTEM_VISION.md) に、その情報設計の原則(形状を運ばず、復元不能な判断材料だけを流通させる)を [INFORMATION_ARCHITECTURE.md](INFORMATION_ARCHITECTURE.md) に記録しています。いずれも構想資料であり、採用時に本書へ昇格させます。

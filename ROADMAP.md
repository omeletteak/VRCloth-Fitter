# VRCloth-Declipper ロードマップ

このドキュメントは VRCloth-Declipper の開発計画を示します。背景にある設計判断と競合分析は [docs/DESIGN.md](docs/DESIGN.md) を、ドキュメント全体の索引は [docs/README.md](docs/README.md) を参照してください。

## 基本理念

- **貫通修正に特化する** — 衣装の体型変換(リターゲティング)は先行ツールの領分。本プロジェクトは「着せた後に残る貫通の自動修正」という、誰も埋めていない後段を担う
- **No Cache** — アバター素体形状を復元しうるデータを保存・出力しない。形状由来の中間データはその場で計算し、その場で捨てる
- **Modular Avatar / NDMF エコシステムとの互換** — Merge Armature による骨格揃えを前提とし、非破壊ワークフローを徹底する
- **MIT ライセンスのオープンソース** — 上記の約束(特に No Cache)をコードで検証可能にする

- [ ] **E2E 検証** — 実アバター+衣装で貫通解消を確認(上記5項目の実機目視をまとめて行う。手順は [docs/E2E_TEST_GUIDE.md](docs/E2E_TEST_GUIDE.md))。**進捗(2026-06-17)**: **計算合否は達成**(Bake 後の公平比較で Preflight 残留 0、実スキニング整合済ユニット 99 緑)。**目視クロステストは延期**: 検証に使った同一素体ベースのペアは Modular Avatar Blendshape Sync が効き、素体シェイプキーを 0 にすると衣装が体型追従変形して貫通自体が消えるため、Run の効果と Sync 変形を切り分けた目視ができない(§1.1 の二重適用回避策も Sync 追従衣装では通用しない)。純粋な目視確認には **Blendshape Sync 無し(体型非追従)の衣装×別体型アバターのペア**が要る。この目視はフェーズ3「二重適用解消」後に素体既定のまま取れるようになる(本ペアはその必要性の実機実証)。**方針**: 計算合否でフェーズ1コアを確定し、目視はフェーズ3後/Sync 無しペアで取り直す(2026-06-17 決定)。**更新(同日)**: その後、自動追従なしの衣装(別作家)×同一トポロジ姉妹のスケール差ペアを Bake 後でクロステストし、薄物(パンツ)の体型差貫通の解消を可視確認(**目視クロステスト第一例成立**)。残留は厚物(靴)の内側壁という偽陽性で可視破綻なし([docs/DETECTION_SEMANTICS.md](docs/DETECTION_SEMANTICS.md))。コアは実機で機能と確認。残: 別トポロジ/偽陽性を含まないケースでの追試

## フェーズ3: 品質向上

- [ ] **カプセル半径の自動推定** — 素体メッシュから半径を推定。**v1 landed**(最近接軸割り当て+パーセンタイル+外れ値ゲート; 胴/首/脚/上腕はアバター別の妥当値へ、胴は固定0.12→約0.06で誤検出消滅)。残課題は当初スキニング基準の半径割り当てを想定したが、E2E ヌルテストで**円断面カプセルは胴・足の形状を半径調整では表せない**と判明。解決はフェーズ1の**メッシュSDF コライダ**へ移譲(本項目はその前段の知見として残す)。カプセルは高速プレビュー用に残す可能性
- [ ] **シェイプキー・ポーズ対応** — 代表ポーズ/シェイプキーでの貫通もまとめて修正。あわせて**適用の二重適用を解消する**: 現状の書き戻し([VRClothMeshApplier](Assets/VRCloth-Declipper/Editor/VRClothMeshApplier.cs))は焼いた座標(シェイプキー変位込み)でベース頂点を置換しブレンドシェイプを複製に残すため、非0のシェイプキー(Modular Avatar の Blendshape Sync で素体に追従)が二重に乗って崩れる。**「焼いた座標で置換」ではなく「元ベース頂点へ補正デルタを加算」**に変えれば静的体型では二重適用が消える(動的に変わる Sync 値の追従が本項の本丸)。**二重適用(静的)landed(2026-06-17)**: Applier を「置換」→「補正デルタ加算」へ(`SkinningMath.WorldDeltaToMeshLocal` 新設、`ClothSnapshot.bakedWorld` でフィット前ワールド座標を保持)。ブレンドシェイプ100%下で補正0適用→再bakeが元キャプチャと一致するE2Eユニットで二重適用解消を固定(EditMode 102緑/1 explicit skip)。**代表ポーズ対応の核「合成」landed(2026-06-18)**: N ポーズの補正を bind 空間の静的デルタ1枚へ束ねる過剰拘束を、蓄積デルタのポーズ掃引(Gauss-Seidel、押し出し単調ゆえ有界収束)で解く純 Core `MultiPoseComposer` をユニット付きで実装(設計は [docs/MULTIPOSE_COMPOSITION.md](docs/MULTIPOSE_COMPOSITION.md))。残: **エディタ glue**(各ポーズへ駆動→Bake→捕捉→適用、BakeMesh×HumanPose の経験的スパイク)・ポーズ較正(目視)・**動的 Sync 追従(本丸)**。**glue 着手前の実験計画 landed(2026-06-19)**: glue の置き場所(エディタ=マージ前プレビュー / NDMF パス=マージ後だがビルド時 T-pose 固定)で別々の壁に当たり、特に**ビルド中のポーズ駆動 Bake** はパス順とは独立した未検証領域。2段スパイク(段階1 エディタで Composer 数値検証→段階2 ビルド時ポーズ駆動 Bake)の計画は [docs/MULTIPOSE_GLUE_SPIKE.md](docs/MULTIPOSE_GLUE_SPIKE.md)。**段階1 glue landed(2026-06-21)**: エディタの **Run Multi-Pose Fit (Spike)** ボタン([VRClothMultiPoseSpike](Assets/VRCloth-Declipper/Editor/VRClothMultiPoseSpike.cs))が代表ポーズへ `HumanPoseHandler` 駆動→Bake 捕捉→`Compose`→`ApplyBindLocalDelta`(置換でなく加算)→各ポーズ残留ログまで配線。数値チェーン(捕捉 skin 行列・収束・二重適用回避)は実 SMR の `MultiPoseRoundTripTests` で固定(EditMode 114 緑)。残: **実アバターでの目視較正**([docs/E2E_TEST_GUIDE.md](docs/E2E_TEST_GUIDE.md) §7.1、muscle 値は経験的叩き台)・**段階2**(ビルド時ポーズ駆動 Bake)・**動的 Sync 追従(本丸)**
- [ ] **縮め/隠しシェイプキーへの堅牢性と原因の名指し** — 衣装の一部を意図的に縮める/隠すシェイプキー(Blendshape Sync 無しの独立トグル)は、畳んだ頂点を体内深部へ押し込む。検出はこれを深い貫通として正しく拾うが、結果は偽RED→当該レンダラーごと skip(安全側だが、同レンダラーの可視部分の実貫通も巻き添えで未修正になりうる)。**この失敗モードは発見が難しい**: 縮んだ見た目が意図的デザインに見えて目視では気づけず、唯一の手掛かりの RED も既定メッセージが「リターゲ級の体型差」と誤った原因を指す。対策は2段: (1)畳み込み(体内深部で密集・三角形が縮退)頂点を検出から除外し可視部のみ対象にする、(2)RED の数値署名(異常な深さ＋侵入率と最大連結パッチが高位で一致＝『深くて一塊』、畳まれた三角形の縮退)から原因を判別し「縮め/隠しシェイプキーの可能性 — Blendshape を確認」と**名指し**して気づけるようにする。**実装 landed(2026-06-18)**: (1) `CollapsedVertices.Find`(縮退三角形=面積≒0 のみに接する頂点を抽出)を `VRClothPenetrationDetector` が hits から除外、(2) `PreflightDiagnostic` に `RedCause`(`CollapsedShapeKey`/`RetargetingClassDifference`)を追加し pipeline 警告・RunLog で名指し。しきい値は暫定(E2E 較正対象)。EditMode 108 件緑。残: **実機 E2E での効き目確認**(左ソックス短縮SKが約53mm内側と誤検出、中立化で解消 — [docs/E2E_TEST_GUIDE.md](docs/E2E_TEST_GUIDE.md) §0.1)。この目視は除外しきい値の較正も兼ねる
- [ ] **偽陽性の整理(可視でない貫通の扱い)** — 検出は拾うが可視面に破綻のない貫通を「直すべき真の貫通」と区別する。代表例は**厚物/二重メッシュ衣装(靴・ブーツ等)の内側壁**が素体に正当接触する型 — 押し出しでは解かず(内側壁を押すと衣装が歪む)、検出から除外するか合否を可視面で取る。類型と合否基準は [docs/DETECTION_SEMANTICS.md](docs/DETECTION_SEMANTICS.md)。実機確認: 2026-06-17 クロステストで靴の内側壁が残留として出たが可視はクリア。**保留(2026-06-18)**: 手法選択そのものが未決の設計判断 — (a) 法線×素体勾配のバックフェース除外、(b) 可視外側面でのみ合否、(c) 厚み基準のいずれか。誤除外は『実貫通の取りこぼし』(最悪方向)に振れるため、暫定既定で進めず判断を先送り。**更新(2026-06-22)**: Mini Stack × SELESTIA クロステストで本偽陽性が再発確認(Necklace 侵入87%/Neck_Belt/Leg_Belt が「高比率×最大連結パッチ小×浅い」=内側壁署名)。**暫定対応 landed(commit `ec355aa`)**: `PreflightDiagnostic` に3つ目の `RedCause.ThickGarmentInnerWall` を追加し、従来一律 `RetargetingClassDifference` と誤ラベルしていた分散・浅い・高比率の RED を「内側壁の偽陽性くさい・要目視・リターゲではない」と名指しへ是正(連結情報がある時=`largestPatchRatio>0` のみ判定するので均一サイズ不適合や三角形なしケースは従来どおり=非破壊。判定は RED 維持=**ラベルのみ**)。ただしこれは表示の是正どまりで、検出は依然内側壁をカウントする。**残=本丸の手法決定+実装**: 上記 **(a) 法線×素体勾配のバックフェース除外**を本命に、内側壁を検出から除外して可視外側面で合否を取る(アクセが正しく GREEN に落ちる)。SELESTIA クロステストが較正の実証ケース
- [ ] **関節部の可変マージン** — 関節をまたぐウェイトを持つ頂点のマージンを増し、曲げ起因の貫通への安価な保険とする(本命は上のポーズ対応。背景は [docs/DEFORMATION_METHODS.md](docs/DEFORMATION_METHODS.md) §5)。**保留(2026-06-18)**: ソルバ全体がスカラー margin 前提(検出・押し出し・再検出の複数箇所)で、頂点別マージン化は加算的(既定 null=現行挙動)に低リスクで可能だが、検出/押し出し/ソルバ署名に波及する最大の変更 + 「関節跨ぎ」定義と増幅率の設計判断を要する。`CollapsedVertices` 同様の Core 分類器(ボーンウェイト+親子マップ→跨ぎ頂点)として実装可能。判断確定後に着手。**深掘り済み(2026-06-18)**: 設計と判断整理は [docs/JOINT_MARGIN.md](docs/JOINT_MARGIN.md)(予防的マージンの特異点、検出に効かせるか押し出しだけかのフォーク、posed capture が既に動くため本命=代表ポーズ対応へ飛ばす方が ROI 高い可能性)

## 凍結: 差分キャッシュの保存・共有

旧ロードマップにあった「差分キャッシュの保存・再利用」「体型差分プリセットの共有」(旧 FUTURE_PLANS.md の構想を含む)は**凍結**します。

- 体型差分の共有という方向は、変換型の先行ツールが既に担っており機能的な空白がない
- そして保存・配布する差分データは原理的に元アバターの体型情報を含み、No Cache 原則と衝突する(詳細は [docs/DESIGN.md](docs/DESIGN.md) §4)
- 量子化やノイズ付加による「安全化」も検討したが、安全性を保証する形式の議論が固まらない限り実装しない。これが No Cache 原則の由来

## 経緯(旧世代の記録)

- **第1世代(ボーンベースフィッティング)**: ボーンスケーリングとアンカー式メッシュ変形を実装したが、メッシュ変形はメッシュ縮みが頻発し設計として破綻、廃止
- **第2世代(ケージ変形構想)**: FFD ケージによる手動変形を構想(フェーズ4に縮小して継承)
- **第3世代(現行)**: 「本質は Merge Armature 後の貫通修正」と再定義し、貫通検出→押し出し→平滑化のパイプラインに一本化
